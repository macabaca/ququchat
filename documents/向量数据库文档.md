# 向量数据库文档（Qdrant）

## 1. 目标与范围

本文档定义 ququchat 项目向量数据的统一结构，作为后续实现与扩展的唯一参照标准。  
当前阶段仅启用原始片段向量检索（raw），预留摘要向量（summary）字段。

---

## 2. Collection 规范

- 向量数据库：Qdrant
- Collection 名称：`chat_segments_v1`
- 距离函数：`Cosine`
- 向量维度：`1024`
- 向量组织方式：Named Vectors（`raw` + `summary`）

### 2.1 向量主结构（Named Vectors）

- `raw`：
  - 维度：`1024`
  - 用途：当前主检索向量（由原始聊天分段文本生成）
- `summary`：
  - 维度：`1024`
  - 用途：预留字段，后续用于摘要检索增强
  - 当前状态：可为空或占位；由 `summary_ready` 标识是否可用

---

## 3. Point 主结构规范

每条向量记录（Point）必须包含以下三部分：

- `id`：Qdrant 点 ID（UUID 字符串）
- `vector`：Named Vectors
- `payload`：业务元数据（用于过滤、回溯、治理）

### 3.1 标准 JSON 结构

```json
{
  "id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "vector": {
    "raw": [/* 1024 维 float */],
    "summary": [/* 1024 维 float，可选/占位 */]
  },
  "payload": {
    "room_id": "room_xxx",
    "segment_id": "seg_xxx",
    "start_seq": 120,
    "end_seq": 135,
    "start_ts": 1773740000,
    "end_ts": 1773740600,
    "message_count": 16,
    "summary_ready": false,
    "embedding_model_raw": "BAAI/bge-m3",
    "embedding_model_summary": "",
    "text_preview": "今天讨论了周末出游安排"
  }
}
```

---

## 4. Payload 字段定义（最终版）

### 4.1 必填字段

- `room_id` (string)
  - 含义：房间 ID
  - 用途：检索时强制过滤，避免跨房间召回
- `segment_id` (string)
  - 含义：业务段 ID（对应 PostgreSQL `chat_segments.segment_id`）
- `start_seq` (int64)
  - 含义：该段起始消息序号
- `end_seq` (int64)
  - 含义：该段结束消息序号
- `start_ts` (int64)
  - 含义：起始时间（Unix 秒）
- `end_ts` (int64)
  - 含义：结束时间（Unix 秒）
- `message_count` (int)
  - 含义：段内消息条数
- `summary_ready` (bool)
  - 含义：摘要向量是否已就绪
- `embedding_model_raw` (string)
  - 含义：raw 向量使用的 embedding 模型名

### 4.2 可选字段

- `embedding_model_summary` (string)
  - 含义：summary 向量模型名
- `text_preview` (string)
  - 含义：仅用于展示/调试的短预览，不参与 embedding

---

## 5. 检索约束（必须遵守）

- 所有检索请求必须包含 `room_id` 过滤条件
- 默认使用 `raw` 向量检索
- `summary_ready=false` 的点不参与 `summary` 通道检索
- 回答生成时优先回溯 PostgreSQL 原始分段文本，不以 `text_preview` 作为证据源

---

## 6. 命名与兼容性约定

- Collection 采用版本命名：`chat_segments_v1`
- 若未来更换 embedding 模型且维度或语义分布变化明显，应新建 `chat_segments_v2`，避免直接覆盖
- `segment_id` 与 `qdrant point id` 必须稳定可重建，确保幂等 upsert
- Qdrant `id` 必须是 UUID 或无符号整数；当前实现统一使用“由 `segment_id` 计算得到的稳定 UUID”
- 同一 `room_id` 下，若已存在相同 `start_seq + end_seq` 的分段，新分段必须跳过（不写 PostgreSQL，不写 Qdrant）

---

## 7. 与 PostgreSQL 对应关系

Qdrant Point 与 PostgreSQL `chat_segments` 一一对应，建议映射：

- `payload.segment_id` <-> `chat_segments.segment_id`
- `payload.room_id` <-> `chat_segments.room_id`
- `payload.start_seq/end_seq` <-> `chat_segments.start_seq/end_seq`
- `payload.start_ts/end_ts` <-> `chat_segments.start_at/end_at`
- `id` <-> `chat_segments.qdrant_point_id`
- `payload.summary_ready` <-> `chat_segments.summary_ready`

---

## 8. 当前阶段执行策略

- `\生成rag` 与 `\添加记忆` 都会生产并写入 `raw` 向量
- `summary` 向量保留结构，不作为召回入口
- 后续摘要功能上线后，按 `summary_ready` 增量补齐，不改主结构
