# 分布式改造计划

## 目标

将当前 WebSocket Hub 从单机广播中心改造为节点级路由模型，使每个后端实例只处理与自己相关的消息，降低 Redis 订阅开销和节点间无效流量。

## 核心设计思想

- 每台后端实例对应一个 `nodeID` 和一个本地 `Hub`
- 每个 WebSocket 连接对应一个 `connID`
- Redis 负责维护“用户/连接 -> 节点”映射
- Pub/Sub 设计为“节点频道”，而不是“用户频道”或“房间频道”
- 节点之间只转发真正需要的消息，最终由本机 `Hub` 负责送到本地客户端

## Redis 数据结构设计

### 1. 连接关系

- `ququchat:ws:user_conns:<userID>` -> set(connID)
  - 记录用户在全局的所有活跃 WebSocket 连接
  - 支持用户多端登录

- `ququchat:ws:conn_hub:<connID>` -> nodeID
  - 记录某个连接属于哪个后端节点

- `ququchat:ws:hub_conns:<nodeID>` -> set(connID)（可选）
  - 记录某个节点当前所有活跃连接
  - 有助于节点级清理和故障恢复

### 2. 用户-节点关系

- `ququchat:ws:user_nodes:<userID>` -> set(nodeID)
  - 记录用户在哪些节点上有活动连接
  - 私聊路由时直接查这个集合

- `ququchat:ws:node_users:<nodeID>` -> set(userID)（可选）
  - 记录某个节点当前在线用户集合
  - 有助于节点重启恢复或统计

### 3. 房间-节点关系

- `ququchat:ws:room_nodes:<roomID>` -> set(nodeID)
  - 记录该房间在多个节点上有哪些在线成员
  - 群消息路由时直接查这个集合

## Pub/Sub 频道设计

### 1. 节点频道

- `ququchat:ws:hub:<nodeID>`
  - 每个节点订阅自己的频道
  - 发送者按目标节点 publish 消息到该频道
  - 仅目标节点会收到消息

### 2. 可选扩展频道

- `ququchat:ws:hub:<nodeID>:user`
- `ququchat:ws:hub:<nodeID>:room`

这两个频道可用于进一步区分同一节点内部的消息类型，但初期不必复杂化，单一节点频道即可。

## 消息流程设计

### 1. 连接建立

节点 `node1` 收到用户 `userA` 的 WebSocket 连接后：

1. 生成 `connID`
2. 写 Redis：
   - `SADD ququchat:ws:user_conns:userA connID`
   - `SET ququchat:ws:conn_hub:connID node1`
   - `SADD ququchat:ws:user_nodes:userA node1`
   - `SADD ququchat:ws:hub_conns:node1 connID`（可选）
   - `SADD ququchat:ws:node_users:node1 userA`（可选）
3. 加载该用户当前所在的房间列表，并对每个 `roomR` 执行：
   - `SADD ququchat:ws:room_nodes:roomR node1`

### 2. 连接关闭

当 `connID` 断开后：

1. 查 `node1 = GET ququchat:ws:conn_hub:<connID>`
2. 删除关联：
   - `SREM ququchat:ws:user_conns:userA connID`
   - `DEL ququchat:ws:conn_hub:<connID>`
   - `SREM ququchat:ws:hub_conns:node1 connID`（可选）
3. 判断 `userA` 在本节点上是否仍有其他连接：
   - 如果没有，则：
     - `SREM ququchat:ws:user_nodes:userA node1`
     - `SREM ququchat:ws:node_users:node1 userA`
4. 判断房间节点关系：
   - 如果 `userA` 在本节点上不再有连接，则检查该用户当前在线的房间列表
   - 对于每个 roomR，本节点若不再有任何在线成员，则：
     - `SREM ququchat:ws:room_nodes:roomR node1`

> 推荐在节点内部维护本地计数，如 `localUserConnCount[userID]` 和 `localRoomMemberCount[roomID]`，避免频繁读写 Redis 并确保断开清理准确。

### 3. 私聊消息路由

假设 `userA` 在 `node1` 给 `userB` 发消息：

1. 查 `targets = SMEMBERS ququchat:ws:user_nodes:userB`
2. 如果 `node1` 在 `targets` 中，则本机优先直接发送给本机连接
3. 对其他目标节点执行 publish：
   - `PUBLISH ququchat:ws:hub:nodeX <payload>`
   - `PUBLISH ququchat:ws:hub:nodeY <payload>`
4. payload 中建议包含：
   - `type`、`target_user_id`、`from_user_id`
   - `room_id`、`message_id`、`origin_node_id`
   - 可选 `target_conn_ids`

### 4. 群消息路由

假设 `userA` 在 `node1` 给房间 `roomR` 发消息：

1. 查 `nodes = SMEMBERS ququchat:ws:room_nodes:roomR`
2. 本机先将消息发送给本机 `roomR` 的在线成员
3. 对其他节点执行 publish：
   - `PUBLISH ququchat:ws:hub:nodeX <payload>`
   - `PUBLISH ququchat:ws:hub:nodeY <payload>`
4. payload 中建议包含：
   - `type`、`room_id`、`from_user_id`
   - `content`、`message_id`、`origin_node_id`

### 5. 节点接收处理

节点 `node2` 订阅 `ququchat:ws:hub:node2`，收到消息后：

1. 解析 JSON
2. 如果 `origin_node_id == node2`，忽略该消息
3. 如果是私聊消息：发送给本机 `target_user_id` 的连接
4. 如果是群消息：发送给本机 `room_id` 的在线成员

## 节点本地状态维护

### 建议维护的本地内存结构

- `localUserConnCount[userID]`：本机上某用户连接数
- `localRoomMemberCount[roomID]`：本机上某房间在线成员数
- `localUserRooms[userID]`：本机上某用户所在房间列表

### 用途

- 断开时判断是否需要从 `user_nodes` 或 `room_nodes` 中移除节点
- 避免同一用户多连接时误删路由记录
- 降低 Redis 访问频次

## 方案优点

- 订阅量低：每个节点只订阅一个频道
- 路由更精确：仅把消息发给目标节点
- 支持用户多端登录
- 相比用户级频道或房间级频道，更适合节点数较少、用户较多的场景

## 风险与注意点

- `user_nodes` 和 `room_nodes` 需要及时清理，否则会造成冗余广播
- 节点异常崩溃时，需要重建本地状态或清理残留 Redis 记录
- 如果节点数量很多，节点级路由仍然可能产生较多目标节点查询开销
- 若用户同时登录多个节点，消息可能会发送到多个节点，需在节点接收端做幂等或重复判断

## 进一步扩展建议

- 使用 `connID` 作为具体连接标识，未来可支持精细路由到单个连接
- 在 `payload` 中携带 `target_conn_ids`，让目标节点直接定位本机具体连接
- 如果群数量爆炸，可考虑把 `room_nodes` 记录为 `room_shard` 或按房间热度分级
- 结合 `Redis TTL` 做辅助清理，防止节点挂掉后 key 长时间残留

## 结论

该方案适合你当前想要的“节点级路由”思路：

- 后端只是把“消息路由到目标节点”
- 每个节点只处理自己相关的消息
- 本机 `Hub` 继续负责最终送达
